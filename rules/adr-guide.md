# ADR ガイド

## 目的

ADR（Architecture Decision Record）は、アーキテクチャに関する決定や長期的な決定を記録するための文書です。
すべての変更を記録することが目的ではありません。
重要な決定の背景、検討した選択肢、採用理由を将来に残すことを目的とします。

## ADR を作成するタイミング

以下のような長期的な影響を持つ決定を行う場合は ADR を作成してください。

- 新しい技術やツールを採用する場合
- 既存の技術やツールを廃止する場合
- 技術やツールを置き換える場合
- アーキテクチャを変更する場合
- ライブラリやフレームワークの選定を変更する場合
- 開発プロセスを変更する場合
- テスト戦略を変更する場合
- デプロイ戦略を変更する場合
- データベース戦略を変更する場合
- プロジェクト構造を変更する場合
- 運用方針を変更する場合

## ADR を作成しない場合

以下のような変更は ADR の対象外です。

- バグ修正
- 小規模な機能追加
- 設定値の調整
- リファクタリング
- コメントの更新
- ドキュメントの更新
- コードスタイルの変更
- テストケースの追加

## ADR のステータス

### Proposed

検討中。まだ決定されていない。

### Accepted

採用済み。現在有効な決定。

### Rejected

検討されたが採用されなかった。

### Superseded

別のADRによって置き換えられた。

## 命名規則

ADR は以下の形式で管理します。

``0001-short-title.md``

例：

```
0001-config-philosophy.md
0002-use-lazy-nvim.md
0003-use-blink-cmp.md
```

## ADRの構成

各ADRは以下の項目を含めてください。

```md
# Title

Status: Proposed | Accepted | Rejected | Superseded

## Context

背景や課題

## Decision

採用する決定

## Consequences

メリット
デメリット
トレードオフ

## Alternatives Considered

```

## レビューのトリガー

以下の場合は ADR の見直しを検討してください。

- 現在の決定の維持コストが高くなった場合
- 前提条件が変化した場合
- 採用した技術のメンテナンスが終了した場合
- 新しい標準機能で代替可能になった場合
- よりシンプルな解決策が利用可能になった場合
- 組織やプロジェクトの要件が変化した場合

## 運用ルール

- ADR は決定時点の判断を記録する。
- 過去の ADR は削除しない。
- 決定が変更された場合は新しい ADR を作成する。
- 古い ADR は Superseded に変更し、置き換えた ADR を参照する。
- ADR は「何を決めたか」だけでなく「なぜ決めたか」を記録する。
